dbt란? ETL vs ELT

기존 (ETL)

데이터가 DB로 들어가기 전에 외부 엔진에서 변환합니다.

Extract -> Transform -> Load

현대적 (ELT)

데이터를 먼저 적재하고 DB의 성능을 활용해 변환합니다.

Extract -> Load -> Transform

dbt는 SQL만 알면 누구나 고성능 데이터 엔지니어링을 할 수 있게 해줍니다.

Materialization 빌드 전략

SQL 결과물을 DB에 어떻게 물리적으로 저장할지 결정합니다.

view

항상 최신 데이터.
조회 시마다 SQL 실행.

table

물리적 테이블로 저장.
빠른 조회, 전체 재생성.

incremental

변경된 데이터만 추가.
빌드 속도 매우 빠름.

ephemeral

DB에 저장 안 함.
CTE(임시)로만 활용.

본 실습은 incremental + append 전략으로 대용량 처리 기법을 배웁니다.

dbt Hooks: 실행 제어

Hook 명칭실행 시점 및 목적
on_run_start전체 빌드 시작 전 1회 실행. 로그 초기화 등.
on_run_end전체 빌드 완료 후 1회 실행. 임시 테이블 정리 등.
pre-hook개별 모델 빌드 전 실행. 시작 로그 기록.
post-hook개별 모델 빌드 후 실행. 권한(Grant) 부여 등.
우리는 pre-hook을 사용하여 빌드 현황을 DB에 실시간으로 남깁니다.

dbt_project.yml: 계층적 설정

모델별 저장 방식과 스키마를 폴더 단위로 일괄 지정합니다.

# dbt_project.yml 핵심 내용 models: edu001: +pre-hook: "{{ log_model_start() }}" stg: +materialized: incremental +schema: stg +tags: ["stg"] marts: +schema: marts +tags: ["marts"]

설정의 핵심

  • +tags: 실행 시 --select tag:stg로 특정 레이어만 구동 가능
  • +schema: dbt가 자동으로 stg_ 접두어를 붙여 스키마를 생성
  • +pre-hook: 모든 모델 실행 전 로그 매크로 호출

    DAG 필수 설정 (Best Practice)

    운영 사고를 막기 위한 3가지 황금 규칙입니다.

    with DAG( 'dbt_daily_flow', is_paused_upon_creation=True, # ① 즉시 실행 방지 catchup=False, # ② 과거 누락분 일괄 자동 실행 방지 default_args={'depends_on_past': False}, # ③ 이전 성공 여부 무관 ) as dag:

    특히 catchup=False는 처음 배포 시 과거 1년치 데이터가 한꺼번에 도는 것을 막아줍니다.

    dbt_daily_flow.py 전체 구조

    전체 파이프라인의 조립도입니다.

    with DAG('dbt_daily_flow', ...) as dag: start = EmptyOperator(task_id='start') # Staging 레이어 그룹 tg_staging = get_dbt_tag_task_group(dag, 'staging_layer', 'stg', 'stg') # Mart 레이어 그룹 tg_marts = get_dbt_tag_task_group(dag, 'marts_layer', 'marts', 'marts') end = EmptyOperator(task_id='end') # 선후 관계 연결 start >> tg_staging >> tg_marts >> end

    Cosmos — dbt 통합의 혁신

    Cosmos는 dbt 프로젝트 폴더를 읽어 Airflow Task를 실시간으로 자동 생성합니다.
